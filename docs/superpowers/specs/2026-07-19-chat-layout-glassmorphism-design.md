# 聊天页面布局与玻璃质感整理设计

## 背景

当前聊天页面同时存在布局与视觉层级问题：消息列表在长对话下会继续撑开页面，导致输入框离开可视区域；大面积渐变、模糊和悬浮效果缺少统一层级；消息、工具调用记录和空白状态也没有形成清晰的阅读顺序。

本次调整保留现有的会话、联网搜索、流式输出、金融工作台和本地历史能力，只改善页面空间分配、视觉一致性与响应式可用性。

## 目标

- 在任何消息长度下，让输入区始终停留在聊天面板底部。
- 让消息区成为唯一的主滚动区域，避免页面整体跟随长回答无限延展。
- 保留玻璃质感，但将其收束为统一、克制且不干扰阅读的视觉语言。
- 清晰区分用户消息、助手消息和工具调用记录。
- 在窄屏下优先保证聊天内容和输入操作可见、可用。

## 非目标

- 不新增页面或路由。
- 不改变聊天请求、联网搜索、历史会话、金融数据或 IndexedDB 数据结构。
- 不替换现有文案、模型能力或金融工作台的信息架构。

## 页面结构

### 桌面端

工作区保持“状态栏 + 聊天面板”的纵向结构，工作区和聊天面板都使用确定的可用高度。

聊天面板分为两个区域：

1. 消息列表占用剩余空间并独立滚动。
2. 输入区固定在面板底部，不参与消息列表滚动。

消息列表保留居中且受最大宽度限制的阅读列。新会话在该区域显示简洁的空状态，而非无目的的大块空白。

### 移动端

小屏幕下，聊天面板仍维持“消息可滚动、输入固定”的结构。侧栏不应再抢占主内容高度；按钮和输入区在窄宽度下自动换行，确保发送操作不被遮挡或压缩。

## 视觉语言

- 保留青蓝到紫色的柔和环境渐变，只作为页面背景。
- 所有玻璃面板统一使用低对比半透明白、细边框、适度模糊和轻阴影。
- 不再为每一个区块叠加强烈的渐变、漂浮或悬浮位移动画。
- 状态栏、侧栏、输入区和工具记录使用同一套玻璃层级；阅读内容优先于装饰效果。
- 焦点、悬停、禁用和错误状态维持足够的对比度，并保留现有键盘焦点可见性。

## 消息与工具记录

- 用户消息置于阅读列右侧，采用一档更明确但低饱和的青紫渐变，宽度受上限约束。
- 助手消息置于左侧，使用更弱的通透背景，正文获得最高对比度。
- 长文本、列表、链接和代码块保持换行与横向溢出保护。
- 工具调用记录降为正文后的紧凑辅助信息块，保留数据来源、错误和链接，但不与回答正文竞争视觉注意力。
- 流式生成的状态提示保持轻量，减少持续动画；遵从系统的减少动态效果偏好。

## 输入与交互

- 输入框随多行内容增长，但设置最大高度；超过上限后仅输入框内部滚动。
- 输入区的联网开关、说明、停止和发送操作按可用宽度自适应排列。
- Enter 发送、Shift+Enter 换行和停止生成保持既有行为。
- 新消息与流式内容更新时，消息列表自行管理滚动位置；不得通过页面整体滚动影响输入区。

## 实施边界

- 主要改动位于全局样式、聊天容器和消息列表组件。
- 必要时为消息列表增加真实容器高度测量，移除固定视口高度假设，保证虚拟列表范围与实际可视区域一致。
- 不改动 API、服务器端逻辑、持久化数据或金融工作台核心流程。

## 验收标准

1. 在现有长对话中，输入区全程位于可视范围内，只有消息列表滚动。
2. 新会话显示有意图的空状态，聊天区域不显得故障或空洞。
3. 用户、助手和工具记录能一眼分辨，并在长内容下保持阅读节奏。
4. 320px 以上窄屏宽度下，不出现横向滚动、被遮挡的发送按钮或被压出屏幕的输入区。
5. 流式生成、停止生成、联网搜索、历史会话与金融模式仍可正常使用。
6. 自动化测试覆盖虚拟列表范围及主要布局断言；构建通过。

## 风险与缓解

虚拟列表原先使用固定高度，改为实际可用高度时可能改变渲染范围。实现时将保持范围计算接口稳定，并为不同容器高度补充测试。玻璃面板的半透明度会影响文字可读性，因此以正文对比度和焦点状态为优先级，而非追求更强模糊效果。
